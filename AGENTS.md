## 🔄 标准工作流

### 五步开发法

```
┌────────────────────────────────────────────────────────────┐
│ Step 1: 需求解析                                            │
│ ─────────────────                                           │
│ • 阅读相关规格文档                                           │
│ • 识别涉及的架构层（Layer 0/1/2）                            │
│ • 评估理解度（<70%必须澄清）                                  │
│ • 输出：需求解析报告                                          │
├────────────────────────────────────────────────────────────┤
│ Step 2: 测试先行（TDD）                                       │
│ ─────────────────────                                       │
│ • 写测试文件（必须先失败）                                    │
│ • 验证测试覆盖场景                                           │
│ • 禁止写空测试或假测试                                        │
│ • 输出：测试文件 + 失败截图                                   │
├────────────────────────────────────────────────────────────┤
│ Step 3: 最小实现                                              │
│ ────────────────                                            │
│ • 写最少代码使测试通过                                        │
│ • 即时质量门禁（每函数检查）                                   │
│ • 不添加未测试的功能                                          │
│ • 输出：实现代码 + 通过截图                                   │
├────────────────────────────────────────────────────────────┤
│ Step 4: 重构优化                                              │
│ ────────────────                                            │
│ • 消除重复代码（DRY）                                         │
│ • 优化命名（意图表达）                                        │
│ • 确保测试仍通过                                             │
│ • 输出：重构后代码                                            │
├────────────────────────────────────────────────────────────┤
│ Step 5: 文档同步                                              │
│ ────────────────                                            │
│ • 更新代码docstring                                          │
│ • 更新相关Markdown文档                                        │
│ • 验证一致性                                                 │
│ • 输出：文档更新 + 最终检查清单                                │
└────────────────────────────────────────────────────────────┘
```

## ✅ 质量门禁

### 代码提交前必须通过

```bash
# 运行全部检查
make check

# 包含：
# ─────────────────────────────
# ✅ 代码格式 (black, isort)
# ✅ 类型检查 (mypy --strict)
# ✅ 静态检查 (flake8, pylint)
# ✅ 安全扫描 (bandit)
# ✅ 单元测试 (pytest, 100%通过)
# ✅ 覆盖率 (≥80%, 新代码100%)
```

### 七层审查清单

详见 [docs/standards/AI_REVIEW_CHECKLIST.md](./docs/standards/AI_REVIEW_CHECKLIST.md)

| 层级 | 检查项 | 关键问题 |
|-----|--------|---------|
| L1 | 基础合规 | 格式/类型/测试通过？ |
| L2 | 代码规范 | 命名/注释/类型注解？ |
| L3 | 测试质量 | 覆盖/独立/有效？ |
| L4 | 安全性 | 注入/密钥/沙盒安全？ |
| L5 | 性能 | 复杂度/资源使用？ |
| L6 | 可维护性 | 重复/圈复杂度？ |
| L7 | 文档同步 | 代码与文档一致？ |

## 🚫 绝对禁止

### AI天性陷阱

```markdown
❌ 禁止虚报进度
   错误: "基本完成了"
   正确: "进度70%，剩余测试和文档"

❌ 禁止修改测试通过
   错误: 测试不过就改测试
   正确: 修改代码使测试通过

❌ 禁止创建版本文件（V2/增强版/enhanced 等）
   错误: 创建 file_v2.py, file_enhanced.py, file_加强版.py
   正确: 直接修改 file.py，在原文件内实现改进
   强制: 所有修复、改进、增强必须在原文件完成，禁止任何形式的版本分叉

❌ 禁止跳过TDD
   错误: 先写代码后补测试
   正确: 先写测试后写代码

❌ 禁止硬编码密钥
   错误: API_KEY = "sk-..."
   正确: API_KEY = os.getenv("API_KEY")

❌ 禁止无验证输入
   错误: open(user_input).read()
   正确: 验证路径在允许范围内
```

### 代码红线

- 测试覆盖率 < 80%
- 函数行数 > 50
- 圈复杂度 > 10
- 重复代码 > 3行
- 无类型注解的公共API
- 使用 eval/exec

## 🎓 常见任务指南

### 添加新功能

```markdown
1. 在对应模块创建测试文件
2. 实现功能代码
3. 确保100%测试覆盖
4. 更新模块README
5. 如修改API，更新API文档
6. 更新CHANGELOG
```

### 修复Bug

```markdown
1. 先写复现测试（测试必须先失败）
2. 修复代码使测试通过
3. 运行全量测试确保无回归
4. 提交信息: "fix(scope): description"
```

### 重构代码

```markdown
1. 确保有完整测试覆盖（重构前）
2. 小步重构，频繁验证
3. 每次变更后运行测试
4. 不改变外部行为
5. 更新相关文档
```

### 添加依赖

```markdown
1. 评估必要性（是否有替代方案）
2. 检查许可证兼容性
3. 添加到 pyproject.toml 正确分类
4. 更新依赖登记表
5. 本地安装验证
6. 运行全量测试
```
