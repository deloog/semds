# SEMDS AI代码审查清单

**版本**: v1.0  
**强制执行**: ✅ 是  
**使用时机**: 每次提交前、代码审查时

---

## 🎯 审查哲学

```
审查不是为了挑错，而是为了确保：
1. 代码符合规格
2. 架构约束被遵守
3. 质量门槛被达到
4. 可维护性被保障
```

---

## 📋 第一层：基础合规检查（自动）

### 代码质量门禁

```bash
# 运行检查脚本
make check

# 或手动运行：
black --check .
isort --check-only .
mypy core/ evolution/ storage/
flake8 core/ evolution/ storage/
bandit -r .
pytest --cov=core,evolution,storage --cov-fail-under=80
```

**门禁项**：
| 检查项 | 阈值 | 失败处理 |
|-------|------|---------|
| 代码格式 | 0个文件需格式化 | 运行 `make format` |
| 类型检查 | 0个错误 | 添加类型注解 |
| 静态检查 | 0个错误 | 修复代码 |
| 安全扫描 | 0个中/高危漏洞 | 修复安全问题 |
| 单元测试 | 100%通过 | 修复代码或测试 |
| 代码覆盖率 | ≥80%（新代码100%） | 补充测试 |

> **覆盖率标准**: 详见 TDD_MANDATE.md 第2.1节
>
> - 新代码: 100%
> - 核心/进化模块: ≥90%
> - 存储模块: ≥85%
> - 其他模块: ≥80%

## 📋 第二层：代码规范检查（手动）

### Python代码规范

```markdown
### 命名规范

- [ ] 类名使用PascalCase
- [ ] 函数/变量使用snake_case
- [ ] 常量使用UPPER_SNAKE_CASE
- [ ] 私有成员使用\_single_leading_underscore
- [ ] 命名清晰表达意图（非a/b/c）

### 类型注解

- [ ] 所有函数参数有类型注解
- [ ] 所有函数返回值有类型注解
- [ ] 复杂类型使用TypeAlias
- [ ] Optional明确标注

### 文档字符串

- [ ] 所有公共类有docstring
- [ ] 所有公共函数有docstring
- [ ] docstring包含Args/Returns/Raises
- [ ] 使用Google风格或numpy风格

### 导入规范

- [ ] 导入按标准库/第三方/内部排序
- [ ] 无未使用的导入
- [ ] 无通配符导入（from x import \*）
- [ ] 无循环导入
```

### 架构合规检查

```markdown
### Layer 0保护

- [ ] 未直接修改core/下的已有文件
- [ ] 文件系统操作通过safe_write
- [ ] 代码执行通过sandbox_execute
- [ ] 版本控制通过version_control接口

### 层间通信

- [ ] Layer 2未直接修改Layer 1内部状态
- [ ] Layer 1未直接修改Layer 0
- [ ] 数据库访问通过storage/模块
- [ ] 跨层调用通过明确定义的接口

### 职责单一

- [ ] 每个类只有一个主要职责
- [ ] 每个函数只做一件事
- [ ] 函数行数<50行（特殊情况<100行）
- [ ] 类方法数<20个（特殊情况<30个）
```

## 📋 第三层：测试质量检查

### 测试完整性

```markdown
### 功能覆盖

- [ ] 所有公开函数/方法都有测试
- [ ] 正常路径有测试
- [ ] 异常路径有测试
- [ ] 边界条件有测试（空、None、极值）

### 测试质量

- [ ] 测试名称清晰描述行为
- [ ] 使用AAA模式（Arrange-Act-Assert）
- [ ] 每个测试只有一个断言主题
- [ ] 使用fixtures共享测试数据
- [ ] 测试之间相互独立

### 覆盖率深度

- [ ] 所有if/else分支已覆盖
- [ ] 所有异常处理已覆盖
- [ ] 所有循环边界已覆盖
- [ ] 新代码覆盖率=100%
```

### 测试代码审查

```python
# ✅ 好的测试示例
def test_calculator_raises_on_division_by_zero():
    """计算器在除零时抛出ValueError。"""
    # Arrange
    calc = Calculator()

    # Act & Assert
    with pytest.raises(ValueError, match="division by zero"):
        calc.divide(10, 0)

# ❌ 审查时拒绝的测试示例
def test_calc():  # 命名不清晰
    c = Calculator()  # 变量名缩写
    assert c.divide(10, 2) == 5  # 无 Arrange/Act/Assert 分区
    # 缺少异常测试
    # 缺少边界测试
```

## 📋 第五层：性能检查

### 性能红线

```markdown
### 算法复杂度

- [ ] 无O(n²)以上复杂度的算法（小数据量除外）
- [ ] 循环内无数据库查询（N+1问题）
- [ ] 无内存泄漏风险（大数据集处理）

### 资源使用

- [ ] 文件句柄正确关闭（使用with语句）
- [ ] 数据库连接正确释放
- [ ] 大对象及时释放引用

### 响应时间

- [ ] API响应<200ms（简单查询）
- [ ] 单元测试<1秒/个
- [ ] 无同步阻塞调用（应使用异步）
```

## 🔄 审查流程

### 自审查（AI提交前）

```
1. 运行 make check
2. 对照本清单逐项检查
3. 修复发现的问题
4. 更新文档
5. 生成审查报告
6. 提交并附上报告
```

### 交叉审查（AI之间）

```
1. 分配审查任务给不同AI
2. 审查AI按清单检查
3. 记录问题和建议
4. 开发AI修复问题
5. 审查AI验证修复
6. 通过/拒绝判定
```
