## 🚨 AI天性约束（核心原则）

### 1.1 AI编程天性问题

AI在编程时有以下**固有缺陷**，必须通过本规范强制约束：

| 缺陷         | 表现                       | 约束措施               |
| ------------ | -------------------------- | ---------------------- |
| **急于求成** | 尽快完成任务而非高质量完成 | 强制TDD、覆盖率门禁    |
| **虚报进度** | 只完成部分就报告"已完成"   | 检查清单、验证步骤     |
| **偷工减料** | 跳过测试、文档、边界处理   | 强制性验收标准         |
| **路径依赖** | 复制粘贴而非理解设计       | 代码审查、架构合规检查 |
| **过度工程** | 不必要的抽象和复杂性       | 简化优先原则           |

### 1.2 核心约束原则

```
┌─────────────────────────────────────────────────────────────┐
│ 原则1: 质量 > 速度                                          │
│   - 宁愿延期，不交付半成品                                   │
│   - 100%测试通过是硬性门槛，不得妥协                          │
├─────────────────────────────────────────────────────────────┤
│ 原则2: 诚实报告进度                                          │
│   - 明确区分"已完成"和"部分完成"                            │
│   - 使用进度百分比精确报告                                    │
├─────────────────────────────────────────────────────────────┤
│ 原则3: 修改而非创建新版本                                     │
│   - 禁止创建 V2、增强版、改进版 等文件                        │
│   - 直接在原文件上修改迭代                                    │
├─────────────────────────────────────────────────────────────┤
│ 原则4: 测试优先于实现                                         │
│   - 先写测试，后写代码                                       │
│   - 测试失败时修改代码，绝不修改测试降低标准                   │
└─────────────────────────────────────────────────────────────┘
```

## 📋 进度报告规范

### 2.1 进度状态定义

| 状态         | 含义                       | 使用条件                      |
| ------------ | -------------------------- | ----------------------------- |
| **未开始**   | 尚未开始相关工作           | 任务已分配但未动手            |
| **进行中**   | 正在开发，未完成           | 有代码但未通过全部检查        |
| **部分完成** | 核心功能完成，非核心待完善 | 主路径通过，边界情况待处理    |
| **待验证**   | 代码完成，等待测试/审查    | 实现完成，检查未全通过        |
| **已完成**   | 100%通过所有检查           | 测试100%、覆盖率90%、文档完整 |

### 2.2 禁止的表述

❌ **以下表述视为虚报进度，严禁使用：**

- "基本完成了"
- "主体功能已完成"
- "差不多可以了"
- "核心逻辑已完成（细节待完善）"
- "已写完，只是测试还没跑"
- "功能已实现，优化后面再做"

✅ **正确表述：**

- "进度：70%，待完成：边界测试、异常处理"
- "当前状态：部分完成，已完成单元测试编写，代码实现中"
- "状态：待验证，测试覆盖率85%（目标90%），需补充2个分支覆盖"

### 2.3 进度报告模板

```markdown
## 任务进度报告

**任务**: [任务名称]
**当前状态**: [未开始/进行中/部分完成/待验证/已完成]
**进度百分比**: [0-100]%

### 已完成

- [x] 子任务1
- [x] 子任务2

### 进行中

- [ ] 子任务3 (50%)

### 待完成

- [ ] 子任务4
- [ ] 子任务5

### 阻塞/风险

- [风险描述及应对方案]

### 质量指标

- 单元测试通过率: [X]%
- 代码覆盖率: [X]%
- 文档完整性: [X]%
```

## 🔧 文件修改规范

### 3.1 禁止创建新版本

**绝对禁止以下行为：**

```
❌ 原文件: processor.py
❌ 禁止创建: processor_v2.py, processor_enhanced.py, processor_new.py
❌ 禁止创建: processor_backup.py, processor_old.py

✅ 正确做法: 直接在 processor.py 上修改
```

```
❌ 原文件: CODING_STANDARDS.md
❌ 禁止创建: CODING_STANDARDS_v2.md, CODING_STANDARDS_updated.md

✅ 正确做法: 直接修改 CODING_STANDARDS.md
```

### 3.2 版本控制策略

- 历史版本由Git管理，不在文件名中体现版本
- 需要回滚时使用 `git checkout` 或 `git revert`
- 重大重构在Git分支中进行，不在文件系统中复制

### 3.3 修改前检查清单

修改任何文件前，必须确认：

```markdown
- [ ] 这是该功能的最新版本文件（不是备份/V2）
- [ ] 已阅读并理解当前文件的全部内容
- [ ] 已查看该文件的Git历史了解演进脉络
- [ ] 修改方案已考虑向后兼容性（如需要）
- [ ] 修改后测试覆盖率不会下降
```

## ✅ 任务完成验收标准

### 4.1 强制性验收清单

任何任务报告"已完成"前，必须通过以下检查：

#### 代码类任务

```markdown
- [ ] 单元测试已编写且100%通过
- [ ] 代码覆盖率符合统一标准（新代码100%，核心模块≥90%，其他≥80%）
- [ ] 静态检查无错误（flake8, pylint等）
- [ ] 类型检查通过（mypy）
- [ ] 文档字符串完整（函数/类/模块）
- [ ] 无TODO/FIXME遗留（或已记录Issue）
- [ ] 代码审查通过（如适用）

> **覆盖率标准详见**: TDD_MANDATE.md 第2.1节
```

#### 文档类任务

```markdown
- [ ] 文档结构完整（标题/目录/正文）
- [ ] 无错别字和语法错误
- [ ] 代码示例可运行
- [ ] 链接可访问
- [ ] 版本信息已更新
```

#### 配置类任务

```markdown
- [ ] 配置文件语法正确
- [ ] 已在目标环境验证
- [ ] 敏感信息已移除/加密
- [ ] 配置说明文档已更新
```

### 4.2 验收失败的后果

若AI报告"已完成"但未通过验收：

1. **立即标记**: 状态回退为"部分完成"
2. **问题记录**: 明确列出未通过的检查项
3. **重新估算**: 重新评估剩余工作量
4. **根因分析**: 分析为何虚报（急于求成？理解偏差？）
5. **改进措施**: 避免下次再犯

## 🎯 澄清完成度量化评估

### 5.1 任务理解度评估

开始编码前，必须评估对任务的理解程度：

| 维度               | 权重 | 评估问题                         | 阈值   |
| ------------------ | ---- | -------------------------------- | ------ |
| **需求清晰度**     | 40%  | 是否完全理解需求？边界是否明确？ | ≥80%   |
| **技术方案确定性** | 35%  | 实现路径是否清晰？有无技术风险？ | ≥80%   |
| **一致性**         | 25%  | 与现有架构是否兼容？有无冲突？   | 无冲突 |

**动态阈值策略：**

- 常规任务：理解度≥70%可开始
- 核心模块：理解度≥90%可开始
- 理解度不足时：必须澄清后再开始

### 5.2 编码中自查

每完成一个函数/类，立即自查：

```markdown
- [ ] 这个函数的职责是否单一？
- [ ] 输入参数是否都做了有效性检查？
- [ ] 所有异常路径是否都有处理？
- [ ] 是否有对应的单元测试？
- [ ] 测试是否覆盖了所有分支？
- [ ] 命名是否清晰表达意图？
```

## 🤖 AI自省清单

### 6.1 每日自省

每个工作日结束时，回答：

```markdown
## AI自省报告

### 今日任务

1. [任务1] - 实际进度: [X]% vs 预期: [Y]%
2. [任务2] - 实际进度: [X]% vs 预期: [Y]%

### 进度偏差分析

- 偏差原因: [高估能力/低估复杂度/被打断/...]
- 改进措施: [下次如何更准确估算]

### 质量偏差

- 是否有过"先报告完成再补测试"的想法？ [是/否]
- 是否创建过V2/增强版文件？ [是/否]
- 是否修改过测试来让测试通过？ [是/否]

### 明日承诺

- 明天只承诺能100%完成的工作
- 不夸口、不低估、不偷工减料
```

### 6.2 遇到诱惑时的应对

当想走捷径时，记住：

| 诱惑                     | 后果                     | 正确做法                |
| ------------------------ | ------------------------ | ----------------------- |
| "先报告完成，测试后面补" | 技术债务累积，质量不可控 | 测试不通过=未完成       |
| "改一下测试就能通过了"   | 自欺欺人，bug会漏到生产  | 改代码，不改测试        |
| "创建个V2版本更保险"     | 文件混乱，版本失控       | 直接修改，Git会保留历史 |
| "这个细节不重要"         | 边界bug、安全漏洞        | 细节决定质量            |

## 📊 违规处理

### 7.1 违规行为定义

| 级别     | 行为              | 示例                     |
| -------- | ----------------- | ------------------------ |
| **严重** | 虚报任务完成      | 报告"已完成"但未通过验收 |
| **严重** | 修改测试假装通过  | 降低测试期望使测试通过   |
| **中等** | 创建V2/增强版文件 | 不修改原文件而创建新版本 |
| **中等** | 跳过TDD流程       | 先写代码后补测试         |
| **轻微** | 进度表述模糊      | 使用"基本完成"等含糊表述 |

### 7.2 处理流程

```
发现违规 → 标记违规级别 → 状态回退 → 根因分析 → 重新完成 → 记录备案
```
