# SEMDS AI开发工作流规范

**版本**: v1.0  
**强制执行**: ✅ 是  
**适用范围**: 所有AI Agent

---

## 🎯 核心原则

```
计划 → 验证 → 实现 → 验证 → 文档 → 交付
```

AI必须遵循**双向验证**模式：

1. **前置验证**：开始实现前验证理解正确
2. **后置验证**：完成后验证符合规格

---

## 📋 任务执行五步法

### 第一步：需求解析（15%时间）

**强制动作**：

```markdown
- [ ] 阅读所有相关规格文档
- [ ] 识别涉及的架构层（Layer 0/1/2）
- [ ] 确认依赖的已有模块
- [ ] 列出需要修改的文件
- [ ] 确认测试要求
```

**产出物**：

```markdown
## 需求解析报告

**任务理解度**: [0-100]%
**涉及的架构层**: [Layer X]
**需要修改的文件**:

1. file1.py - 修改原因
2. file2.py - 修改原因

**依赖模块**:

- module_a (已存在)
- module_b (需要新建)

**风险评估**: [低/中/高]

- 风险点: ...
- 应对措施: ...
```

**理解度<70%时禁止开始编码**

---

### 第二步：测试先行（25%时间）

**TDD强制流程**：

```python
# 1. 先写测试文件（必须失败）
# tests/test_new_feature.py

class TestNewFeature:
    def test_should_handle_normal_case(self):
        """测试正常情况。"""
        result = new_feature(valid_input)
        assert result == expected_output

    def test_should_raise_on_invalid_input(self):
        """测试无效输入抛出异常。"""
        with pytest.raises(ValueError):
            new_feature(invalid_input)

# 2. 运行测试确认失败
# pytest tests/test_new_feature.py -v
# 必须看到 FAILED
```

**禁止**：

- 不要写空测试（`pass`或`assert True`）
- 不要写永远通过的测试
- 不要写与实现无关的测试

---

### 第三步：最小实现（35%时间）

**实现原则**：

```markdown
1. 只写使测试通过的最少代码
2. 不添加未测试的功能
3. 不优化未成为瓶颈的代码
4. 保持函数单一职责
```

**质量门禁**：

```bash
# 每完成一个函数，立即运行：
pytest tests/test_new_feature.py -v
mypy src/new_module.py
black --check src/new_module.py
```

**失败时**：

- 测试失败 → 修改实现代码
- 类型检查失败 → 添加类型注解
- 格式检查失败 → 运行 `black src/`

---

### 第四步：重构优化（15%时间）

**重构清单**：

```markdown
- [ ] 消除重复代码（DRY）
- [ ] 简化复杂表达式
- [ ] 提取 Magic Number 为常量
- [ ] 优化命名（变量/函数/类）
- [ ] 添加文档字符串
- [ ] 检查圈复杂度（<10）
```

**重构后必须**：

```bash
# 1. 全量测试通过
pytest tests/ -v

# 2. 覆盖率不下降
pytest --cov=src --cov-fail-under=80

# 3. 类型检查通过
mypy src/
```

---

### 第五步：文档同步（10%时间）

**同步清单**：

```markdown
- [ ] 代码中的docstring已更新
- [ ] 架构图已更新（如涉及架构变更）
- [ ] API文档已更新（如修改API）
- [ ] 变更日志已记录
- [ ] 相关规格文档已更新
```

**禁止交付**：

- 代码与文档不一致
- 未说明的breaking change
- 未记录的重要决策

## 🔄 迭代模式

### 小步快跑原则

```
每次迭代 <= 50行代码变更
每次迭代 <= 2小时
每次迭代必须通过全部检查
```

### 迭代检查点

```markdown
## 迭代检查点

**变更范围**: [文件列表]
**变更行数**: [X] 行
**测试新增**: [X] 个
**测试通过率**: [X]%
**覆盖率**: [X]%

**自检结果**:

- [ ] 代码符合编码规范
- [ ] 测试100%通过
- [ ] 类型检查通过
- [ ] 文档已同步
```

## 🚫 禁止模式

### 禁止的交付物

```markdown
❌ 未通过测试的代码
❌ 无类型注解的公共API
❌ 覆盖率<80%的模块
❌ 包含TODO的代码（除非是计划内的）
❌ 重复代码（复制粘贴>3行）
❌ 嵌套超过4层的代码
```

### 禁止的工作方式

```markdown
❌ 同时修改多个不相关功能
❌ 修改超过5个文件的大提交
❌ 先写代码后补测试
❌ 修改测试使测试通过
❌ 跳过代码审查直接交付
```

```markdown
- [ ] 安全扫描通过（bandit）

### 文档

- [ ] 代码docstring完整
- [ ] 用户文档已更新
- [ ] API文档已更新
- [ ] 变更日志已记录

### 架构

- [ ] 符合三层架构约束
- [ ] 不违反Layer 0保护
- [ ] 层间通信符合规范

### 提交

- [ ] 提交信息符合规范
- [ ] 提交粒度合适（<5文件）
- [ ] 无敏感信息泄露
```

## 🎓 最佳实践

### 代码审查对话模板

```markdown
**AI提交**:
"已完成[模块]实现，请审查。"

**审查员反馈**:
"[具体问题]，请修改。"

**AI回应**:
"已修改：[修改说明]。验证：[验证结果]。"
```

### 遇到困难时

```markdown
1. 明确描述问题
2. 提供已尝试的方案
3. 请求具体指导（而非泛泛求助）

❌ "这个有问题，怎么解决？"
✅ "在实现[功能]时遇到[具体错误]，已尝试[A]和[B]，但[C]仍失败，请问[D]是否正确做法？"
```
