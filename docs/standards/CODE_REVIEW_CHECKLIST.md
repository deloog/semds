**版本**: v1.0

## 1. 审查原则

### 1.1 审查者职责

- **质量把关**: 确保代码符合项目标准
- **知识传递**: 帮助作者改进代码质量
- **安全审计**: 识别潜在安全风险
- **架构守护**: 维护架构规范

### 1.2 审查态度

```
❌ 禁止：
- 草率LGTM（Looks Good To Me）
- 只挑毛病不提供建议
- 人身攻击

✅ 应该：
- 逐行审查关键逻辑
- 提供建设性反馈
- 认可好的实践
```

## 3. 审查流程

### 3.1 审查步骤

```
Step 1: 理解变更
  - 阅读PR描述
  - 理解变更目的
  - 确认关联Issue

Step 2: 自动化检查
  - 确认CI全部通过
  - 检查覆盖率报告
  - 检查安全扫描

Step 3: 架构审查
  - 检查分层正确性
  - 检查Layer 0保护
  - 检查接口设计

Step 4: 代码审查
  - 逐文件审查
  - 关注关键逻辑
  - 检查测试质量

Step 5: 安全审查
  - 检查安全敏感代码
  - 验证输入处理
  - 检查密钥使用

Step 6: 文档审查
  - 检查docstring
  - 检查文档同步
  - 检查CHANGELOG

Step 7: 总结反馈
  - 分类问题（阻塞/建议）
  - 提供改进建议
  - 确认批准或要求修改
```

### 3.2 审查评论规范

```python
# 阻塞问题（必须修复）
🚫 [Blocking] 此处存在SQL注入风险，请使用参数化查询

# 重要建议（强烈建议修复）
⚠️ [Important] 建议添加输入验证，避免潜在异常

# 一般建议（可选）
💡 [Suggestion] 此处可以提取常量提高可读性

# 肯定
✅ 很好的错误处理！
```

### 3.3 审查结论

| 结论 | 条件 | 操作 |
|------|------|------|
| **批准** | 无阻塞问题，CI通过 | Merge允许 |
| **有条件批准** | 仅非阻塞建议 | 作者可选择性修复后Merge |
| **要求修改** | 有阻塞问题 | 必须修复后重新审查 |
